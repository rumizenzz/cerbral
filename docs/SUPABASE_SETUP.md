# Supabase setup for Cerbral auth

Cerbral uses Supabase as a thin auth layer only — it sends magic-link
emails, runs the GitHub OAuth handshake, and stores a minimal profile
row per user. It never stores sessions, messages, embeddings, or any
other user content. Those live in the user's own GitHub brain repo and
local filesystem.

## One-time project setup

### 1. Create the project
- New Supabase project at [supabase.com](https://supabase.com).
- Region: whatever is closest to your landing-page CDN (Netlify is
  Cloudflare-backed, so any region is fine).
- Strong DB password — save in `~/.cerbral-secrets.env` as
  `SUPABASE_DB_PASSWORD` for later migrations.

Copy these to `site/config.js` (from `site/config.sample.js`):

```js
window.CERBRAL_CONFIG = {
  SUPABASE_URL: 'https://<project-ref>.supabase.co',
  SUPABASE_ANON_KEY: '<anon public key from Settings → API>',
};
```

### 2. Custom SMTP via Resend
So sign-in emails come from `auth@cerbral.com`, not from Supabase's
shared sender. In Supabase → **Project Settings → Auth → SMTP Settings**:

| Field        | Value                                     |
| ------------ | ----------------------------------------- |
| Enable       | On                                        |
| Host         | `smtp.resend.com`                         |
| Port         | `465`                                     |
| Username     | `resend`                                  |
| Password     | `<your Resend API key from ~/.cerbral-secrets.env>` |
| Sender name  | `Cerbral`                                 |
| Sender email | `auth@cerbral.com` (must be a verified sender in Resend) |

### 3. Redirect URLs
**Project Settings → Auth → URL Configuration**:

- **Site URL**: `https://cerbral.com` (or your Netlify URL during dev)
- **Redirect URLs** (add all):
  - `https://cerbral.com/*`
  - `http://localhost:5173/*` (for local dev)
  - `cerbral://auth/success` (desktop deep link; gets used from /welcome)

### 4. Email template for "Magic Link"
**Authentication → Email Templates → Magic Link**. Replace the body with
the Cerbral-branded template committed at
[`emails/auth_magic_link.html`](../emails/auth_magic_link.html) and
[`emails/auth_magic_link.txt`](../emails/auth_magic_link.txt).

The template uses one Supabase variable:
- `{{ .ConfirmationURL }}` — the single magic link that completes sign-in.

The 6-digit code screen (shown when the email is opened on a different
browser than where sign-in started) is generated by the cross-device
pairing flow — it is **not** inserted into the email itself.

### 5. GitHub OAuth provider
**Authentication → Providers → GitHub → Enable**.

You'll need a GitHub OAuth App:
1. GitHub → Settings → Developer settings → OAuth Apps → **New OAuth App**
2. **Application name**: `Cerbral`
3. **Homepage URL**: `https://cerbral.com`
4. **Authorization callback URL**: the value Supabase shows you in the
   provider settings (of the form
   `https://<project-ref>.supabase.co/auth/v1/callback`).
5. Copy the **Client ID** and a newly generated **Client secret** into
   Supabase's GitHub provider settings.
6. In Supabase, set **Scopes** to `repo user:email` — this lets Cerbral
   create the user's private `cerbral-brain` repo on their behalf.

### 6. `profiles` table + RLS
Open **SQL Editor** and run:

```sql
create table if not exists public.profiles (
  user_id       uuid primary key references auth.users(id) on delete cascade,
  email         text not null,
  github_handle text,
  brain_repo_url text,
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);

alter table public.profiles enable row level security;

create policy "self-read"   on public.profiles for select using (auth.uid() = user_id);
create policy "self-upsert" on public.profiles for insert with check (auth.uid() = user_id);
create policy "self-update" on public.profiles for update using (auth.uid() = user_id);
```

No extra columns — Cerbral Cloud intentionally knows almost nothing
about the user.

### 7. Rate limiting
**Authentication → Rate Limits**: defaults are fine. Most relevant:
- `Email OTPs`: 30/hour per IP (Supabase default is good).

## Cross-device pairing (optional, v1.1)

The sign-up flow today assumes the user clicks the magic link on the
same browser where they entered their email. For users who open the
email on a phone but started signup on their laptop, the flow still
works (they just finish on the phone). A nicer UX — "enter this 6-digit
code on the original browser" — requires a small edge function and a
pending-auth table.

When you're ready to add it, run:

```sql
create table public.pending_auth (
  id               uuid primary key default gen_random_uuid(),
  email            text not null,
  requester_cookie text not null,
  pairing_code     text,
  consumed_at      timestamptz,
  created_at       timestamptz default now(),
  expires_at       timestamptz default now() + interval '15 minutes'
);
alter table public.pending_auth enable row level security;
create policy "anyone-insert"  on public.pending_auth for insert with check (true);
create policy "anyone-read-id" on public.pending_auth for select using (true);
create policy "anyone-update"  on public.pending_auth for update using (pairing_code is null);
```

Then deploy the two edge functions in `supabase/functions/auth-start`
and `supabase/functions/claim-pairing` (code lives in this repo once
added — they aren't scaffolded yet; see the roadmap).

## Local dev

```bash
cd cerbral-public
cp site/config.sample.js site/config.js
# edit site/config.js with your Supabase URL + anon key
npx serve site
```

Then open `http://localhost:3000/signup` and try the flow. Your email
will arrive via your Resend → Supabase SMTP configuration.

## What Cerbral Cloud stores per user

Exactly this and nothing more:

| Column            | Purpose                                              |
| ----------------- | ---------------------------------------------------- |
| `user_id` (uuid)  | Supabase-managed identity                            |
| `email`           | Where to send future magic links                     |
| `github_handle`   | Which account owns the brain repo                    |
| `brain_repo_url`  | URL of the user's private `cerbral-brain` repo       |

No sessions. No messages. No model weights. No embeddings. No telemetry.

If the user ever wants to fully delete their Cerbral Cloud footprint,
they delete their `profiles` row → Supabase cascades to `auth.users` →
their record is gone. Their brain repo remains on their GitHub account
(still theirs, Cerbral never had read access to its contents).
